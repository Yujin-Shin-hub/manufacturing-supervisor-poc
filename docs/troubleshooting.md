## 1. EventBus를 도입한 이유 (LINA에서는 안 썼는데 여기서는 쓰는 이유)

**상황** — LINA에서는 SSE 이벤트를 FastAPI 핸들러가 응답 커넥션에 직접 yield했다.
같은 방식을 여기에도 쓸 수 있는지 검토.

**판단** — 인프로세스 pub/sub(EventBus, asyncio.Queue 기반)을 사이에 둔다.
Supervisor·Worker는 `bus.emit()`만 호출하고, 콘솔 로거와 SSE가 각자 구독한다.

**근거** — 두 시스템은 이벤트의 생산자-소비자 관계가 다르다.

| | LINA | 이 프로젝트 |
|---|---|---|
| 스트림 범위 | 요청 단위 (질문→답변) | 실행(run) 단위 |
| 생산자:소비자 | 1:1, 같은 HTTP 요청 수명 안 | 1:N, 별개 커넥션·별개 수명 |
| 소비자 | 질문 보낸 브라우저뿐 | SSE 대시보드 + CLI 콘솔 로거 (+ 향후 센서 트리거) |

구체적으로: (1) 실행 트리거(`POST /run`)와 관찰(`GET /events`)이 분리돼 있어
Supervisor가 "누가 듣는지" 몰라야 결합이 안 생긴다. (2) 같은 이벤트를 콘솔과 화면이
받아야 하는데 버스 없이 하면 emit 지점이 이원화되어 로그와 UI가 어긋날 수 있다.
(3) 스코프 가드 "대시보드가 죽어도 CLI는 독립 동작"은 HTTP 응답에 직접 yield하는
구조로는 성립 불가(CLI엔 HTTP가 없다). (4) 재접속 시 Last-Event-ID 이후 재전송은
seq 순 이벤트 버퍼가 필요하고, 그 버퍼의 자리가 버스다.

**한 줄** — LINA는 요청 범위 스트림이라 직접 yield가 맞았고, 여기는 실행을 바깥에서
여러 소비자가 관찰하는 구조라 발행-구독 분리가 필요했다.

## 2. SSE를 선택한 이유 (폴링·새로고침 대비)

**상황** — MES 현황판은 보통 새로고침/폴링인데 굳이 SSE가 필요한가.

**판단** — 실행 과정 스트림은 SSE, 정적 조회(`/api/*`)는 일반 REST로 이원화.

**근거** — MES 현황판이 보여주는 건 상태(state) 스냅샷이라 30초 전 데이터여도
의사결정이 안 바뀐다 → 폴링이 합리적. 이 대시보드가 보여주는 건 진행 중인
과정(trace)이고, agent_start→tool_call→agent_end가 1초 안에 몰아치는 버스트라
1초 폴링이면 애니메이션이 뭉개지고 고빈도 폴링이면 유휴 시간 낭비다.
클라이언트도 EventSource 내장(자동 재접속 포함)이라 폴링의 since 인덱스 관리보다
오히려 단순하다. 유사 사례: CI/CD 실행 로그(스트리밍) vs ERP 현황판(조회).

**한 줄** — 상태 조회와 실행 추적은 데이터 성격이 달라서 전송 방식도 다르게 골랐다.

## 3. API 키 하드코딩 유출 사고 → .env 체계 전환

**상황** — 초기 프로토타입 노트북에 OpenAI 키가 하드코딩된 채 레포에 포함됐다.

**해결** — 키 즉시 폐기(revoke), 코드에서 `os.getenv()`로 전환, `.env` +
`.gitignore` 체계 도입, `.env.example`로 필요 변수만 문서화.
CLAUDE.md 절대 규칙 1로 승격시켜 재발 방지를 구조화했다.

**근거/교훈** — 노트북은 출력 셀에도 비밀값이 남을 수 있어 커밋 전 `sk-` 문자열
검색을 습관화한다(conventions.md). 히스토리에 남은 키는 삭제 커밋으로 지워지지
않으므로 폐기가 유일한 복구다.

**한 줄** — 키 유출은 지우는 게 아니라 폐기하는 것이고, 재발 방지는 습관이 아니라
구조(규칙+템플릿)로 만든다.

## 4. Windows에서 `pip install -r` UnicodeDecodeError (cp949)

**상황** — requirements.txt에 한글 주석을 넣자 Windows에서
`UnicodeDecodeError: 'cp949' codec can't decode byte ...` 발생.

**해결** — requirements.txt를 영문 주석으로 전환.

**근거** — 파일은 UTF-8인데 한국어 로케일 Windows의 pip이 cp949로 디코드해서 충돌.
파이썬 소스는 기본 UTF-8로 읽으므로 코드·md의 한글은 무관하고, pip이 직접 읽는
파일만 ASCII로 유지하면 된다.

**한 줄** — pip이 읽는 파일(requirements/constraints)에는 한글 주석을 넣지 않는다.

## 5. Playwright UI 감사에서 나온 P1 수정 3건 (2026-07-14)

**상황** — Playwright로 대시보드를 실제 구동·조작해 감사한 결과 P1 3건:
(1) run 종료 시 승인 모달 12개가 리포트 드로어를 덮어 "나중에 처리"를 12번 눌러야 함,
(2) 리스크 추이 차트가 600×120px 기본값에 고정돼 패널 절반이 빈 공간,
(3) 핵심 추천 액션 표(1,198px)가 드로어 폭(671px)을 넘어 결론 컬럼(expected_effect)이
스크롤 전에는 보이지 않음.

**해결** —
(1) 오버레이 우선순위 규칙: 드로어가 열려 있으면 모달을 보류하고(App.vue computed),
"나중에 처리"는 모달 큐 전체를 비운다 — 건들은 PENDING으로 제안 큐·알림센터에 남는다.
(2) 원인은 `v-else`로 늦게 렌더되는 chartHost에 ResizeObserver를 `onMounted`에서만
붙인 것 — ref는 mount 시점에 null이라 observer가 영영 붙지 않았다. `watch(chartHost)`로
observe 대상을 갱신해 해결.
(3) A안 채택: `run_end`에 `key_actions` 구조화 배열을 추가하고(수치는 dispatching 근거 표
값의 타입 변환만), 드로어가 markdown의 해당 절(`<!-- key_actions:start/end -->` 마커)을
고정 컴포넌트(KeyActionsPanel)로 대체 렌더링. CLI·README는 markdown 표를 그대로 쓴다.

**근거** — (3)에서 서버가 HTML을 markdown에 심는 B안은 단계 8에서 HTML 카드→markdown
표로 되돌린 이력(CLI 호환)과 재충돌하고, 프론트가 markdown 표를 파싱하는 C안은 컬럼
변경에 취약하다. 구조화 데이터 + 표시 전용 컴포넌트가 "수치는 코드가 계산, 프론트는
표시만" 원칙과 정확히 일치한다. 같은 감사에서 KPI CRITICAL(전체 CSV 집계 27)과 리포트
(score_delay_risk 집계 18)의 불일치도 발견 — `GET /api/risk-summary`로 KPI가 Worker와
동일한 tool 집계를 쓰게 해 정합시켰다.

**한 줄** — 늦게 렌더되는 ref에 observer는 mount가 아니라 watch로 붙이고,
화면과 리포트가 같은 숫자를 말하려면 같은 tool 함수를 바라봐야 한다.
